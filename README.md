# CEO Skill

> 面向 AI Agent 的架构级任务调度技能——以 CEO 视角梳理全貌、拆分模块、创建 Agent Teams 并行开发、审核验收。

## 为什么需要这个 Skill？

当 AI Agent 面对复杂项目时，常见的失败模式是：

- **上手就写代码**：没想清楚就开干，做到一半发现架构不对
- **单线程推进**：一个人扛所有模块，效率低下
- **模块间断裂**：多个模块各自完成，拼在一起时数据流不通

CEO Skill 解决的核心问题是：**怎么把一个复杂系统拆成多个模块并行开发，拼回来时还能跑。**

## 核心信条

**想清楚再动手，拆清楚再分工，验清楚再收工。**

## 三条直觉

1. **接缝比模块危险**：模块内部 Agent 自己搞定，出事的地方永远是模块间数据流断裂。拆任务时注意力重心放在"数据怎么在模块间流"
2. **隐含假设最致命**：你觉得"当然"的事就是 Agent 一定会猜错的事。写规格时反向审视——从 Agent 视角看文档，拼不出完整链路的地方就是"当然"在作祟
3. **单元测试全绿是危险的安心感**：验收先跑端到端链路，模块测试再绿也不算数

## CEO 定位

**你想，他们干。你审，他们修。你兜底，但不代劳。**

| 阶段 | CEO 角色 | 具体动作 |
|------|---------|---------|
| 架构 | 设计师 | 画出全貌、焊死接缝、写好调度文档 |
| 开发 | 调度员 | 不干预技术路径，确保团队有清晰的协作上下文 |
| 审查 | 质检员 | 跑端到端链路，对照规格逐项审核 |
| 修复 | 项目经理 | 把问题分回原 Agent 修，不自己上手 |
| 兜底 | 缝补者 | 只有小问题或团队无法处理时才亲自缝补 |

## 执行流程

### Phase 1: 想清楚

1. 画出数据流图：从数据源头到最终消费，每经过一个模块就是一次"接缝"
2. 焊死每个接缝：上游输出精确结构、下游期望精确结构、谁负责转换
3. 识别不可变约束
4. 反向审视：假装自己只看调度文档，能不能拼出完整数据流

### Phase 2: 拆清楚

1. 写调度文档（使用 `references/dispatch-template.md` 模板）
2. 调度文档是团队的共享作战地图，包含：
   - 任务边界 + 接口契约 + 接缝定义
   - 协作规则：依赖等待、进度汇报、冲突处理、数据对接
3. 启动 Agent Teams

### Phase 3: 验清楚

1. 先跑端到端链路
2. 逐 Agent 审核
3. 问题分回原 Agent
4. 只有无法分配的问题才亲自缝补

## 调度文档模板

`references/dispatch-template.md` 提供了团队调度文档的标准结构，包含：

- **项目现状**：已完成部分、存在问题、数据源
- **不可变约束**：所有 Agent 必须遵守的硬性规则
- **协作规则**：依赖与等待、进度汇报、冲突处理、数据对接
- **团队定义**：每个 Agent 的边界、背景、规格、对外接口
- **接口契约与接缝定义**：模块间的数据流焊死表
- **进度追踪**：实时状态更新

## 适用场景

- 项目涉及 3 个以上模块需要并行开发
- 系统需要重构且模块间有数据依赖
- 从零搭建多子系统协作的项目
- 任何"一个人搞不定、得拆分工协作"的场景

## 起源

本技能诞生于 2026 年 5 月一次洛克王国 PVP 沙盒系统的 4Agent 并行重构实践。4 个 Agent 各自单元测试全绿，但拼在一起时数据流断裂（SkillUse 字段全空、克制表双源）。

复盘发现根因不是 Agent 能力不足，而是调度文档只定义了"每个模块做什么"，没定义"数据在模块间怎么流"——接缝成了三不管地带。

本技能的目标：**让下次不再踩同一个坑。**
